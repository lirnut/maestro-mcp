# ADR-0003: Maestro v2 — Token Economy, Refactoring & Client-Aware Unification

**Date:** 2026-03-07
**Status:** Accepted
**Authors:** Rômulo + Claude (Opus 4.6, Claude.ai session)
**Supersedes:** Partially supersedes ADR-0001 (agent routing), ADR-0002 (single-shot dispatch)

---

## Context

Maestro has grown from a simple SSH relay into a critical infrastructure control plane — 2,195 lines in `server.py`, 20 MCP tools, OAuth, transfer relay, and a multi-agent orchestra. It serves two fundamentally different clients through the same process:

- **Claude.ai** (phone/iPad/web) → HTTPS → Cloudflare Tunnel → localhost:8222
- **Claude Code** (Apollyon) → HTTP → localhost:8222

Both get identical treatment. This creates compounding friction:

1. **Token waste:** Polling loops, verbose responses, redundant schema — burning ~10,000+ tokens/session on overhead.
2. **UI hangs:** `block_timeout=120` outlives Claude.ai's ~60-90s frontend timeout. The user sees Schrödinger's Session — task simultaneously alive and dead.
3. **Scope creep:** Dispatched agents receive vague prompts, refactor code they weren't asked to touch, and inflate output that flows back into the orchestrator's context.
4. **Monolithic code:** Five distinct domains in one file. Dispatching "fix the transfer relay" to Claude Code exposes the entire 2,200-line file to scope creep.

## Decision

We will execute a phased refactoring of Maestro across three dimensions: **token economy**, **code architecture**, and **client-aware behavior**. The unifying principle:

> **Don't expose controls to the client that the client shouldn't be deciding.**

## Part I: Token Economy

### 1.1 Polling Overhead → Server-Enforced Cooldown

**Problem:** The orchestrator polls compulsively. `wait=0` prevents UI hangs but invites spam. Each poll round-trip costs ~400 tokens.

**Solution:** Server-side poll cooldown per task. `agent_poll` tracks `last_polled_at` on each `TaskState`. Polls within the cooldown window get a minimal rejection:

```python
POLL_COOLDOWN = 30  # seconds

async def agent_poll(task_id: str) -> str:
    ts = TASK_REGISTRY.get(task_id)
    if ts.status == "running" and ts.last_polled_at:
        since_last = (now - ts.last_polled_at).total_seconds()
        if since_last < POLL_COOLDOWN:
            return json.dumps({"status": "cooldown", "retry_after": round(POLL_COOLDOWN - since_last)})
    ts.last_polled_at = now
    # ... existing logic
```

Cooldown is adaptive by agent type: 10s for exec/script, 20s for gemini_research, 45s for claude/codex.

Cooldown bypassed for completed tasks — no point delaying a ready result.

**Additionally:** Stable result paths (`{task_id}_result.json` symlink) enable relay-based collection — the orchestrator pulls results via `bash_tool` + curl, eliminating polling entirely for batch work. The container retries in a sleep loop; zero LLM tokens spent waiting.

### 1.2 Response Verbosity → Lean JSON

| Cut | Savings |
|---|---|
| Remove `poll_with` / `log_with` hints from responses | ~100 bytes/dispatch |
| Minimal "still running" JSON: `{"status":"running","elapsed":45}` | ~100 bytes/poll |
| Reduce `MAX_INLINE_OUTPUT` from 4000 → 1500 chars | ~2,500 bytes/result |
| Trim `_build_instructions()` to <300 chars (host list + behavioral rules only; tool catalog is redundant with MCP schema) | ~900 bytes/session |
| Compress all tool docstrings to one-liners | ~7,000 bytes/session |

### 1.3 Schema Parameter Removal

Parameters the client shouldn't decide are removed from MCP tool schemas entirely. The server decides internally, based on client classification (see Part III).

**Removed from all client-facing schemas:**
- `block_timeout` — server picks based on client profile and agent type
- `timeout` — server uses internal defaults (300s exec, 600s agents)
- `wait` on `agent_poll` — always returns immediately (subject to cooldown)
- `max_budget_usd` on claude tools — don't volunteer spending estimates to Anthropic when their CLI billing isn't transparent

**Before:**
```
claude_dispatch(host, prompt, working_dir, max_budget_usd, allowed_tools, timeout)
maestro_exec(command, host, cwd, sudo, timeout, block_timeout)
agent_poll(task_id, wait)
```

**After:**
```
claude(host, prompt, working_dir, allowed_tools)
exec(command, host, cwd, sudo)
poll(task_id)
```

### 1.4 Agent Scope Discipline

All dispatches are prepended with a scope-guard prefix:

```
SCOPE CONSTRAINTS (non-negotiable):
1. ONLY modify files and code directly related to the task below.
2. Do NOT refactor, clean up, or improve code outside the task scope.
3. Do NOT run tests unless explicitly asked.
4. Do NOT write or update documentation unless explicitly asked.
5. When done, output ONLY: files changed + one-sentence summary per file.
6. If the task is ambiguous, do the MINIMUM viable interpretation.

TASK:
```

Configurable via `scope=` parameter: `"strict"` (default), `"loose"`, `"none"`.

## Part II: Code Architecture

### 2.1 Module Extraction (God Object → Package)

```
maestro-mcp/
├── maestro/
│   ├── __init__.py
│   ├── config.py          # MaestroConfig dataclass, all constants
│   ├── hosts.py           # HostConfig, _load_hosts, resolution
│   ├── transport.py       # SSH: ControlMaster, retry, _async_run
│   ├── local.py           # Local execution: _local_run, _local_script
│   ├── tools/
│   │   ├── __init__.py
│   │   ├── fleet.py       # exec, script, read, write, status
│   │   └── orchestra.py   # Agent dispatch, TaskState, auto-promote, poll
│   ├── relay.py           # Transfer relay HTTP endpoints
│   ├── client.py          # Client classification + profiles (Part III)
│   └── oauth.py           # MaestroOAuthProvider (existing)
├── server.py              # Entry point: wiring only (~100 lines)
├── oauth_rewrite.py       # ASGI middleware (unchanged)
└── tests/
```

Each module has zero implicit cross-dependencies. `transport.py` doesn't know about MCP tools. `orchestra.py` calls `transport.run()`. This makes dispatched refactoring safe — "Edit only `maestro/relay.py`" is an enforceable scope boundary.

### 2.2 Tool Consolidation (20 → ~10)

**Execute/Dispatch merge:** With `block_timeout` removed from schemas, the distinction vanishes. One tool per agent:

| Before (8 tools) | After (3 tools) |
|---|---|
| `codex_execute` + `codex_dispatch` | `codex` |
| `gemini_analyze` + `gemini_execute` + `gemini_dispatch` | `gemini` (with `mode=analyze\|execute\|research`) |
| `claude_execute` + `claude_dispatch` | `claude` |

**Background command merge:** `maestro_bg` + `maestro_bg_log` are subsumed by auto-promote on `exec`/`script`. Remove both.

**Rename:** Drop `maestro_` prefix. MCP namespaces by server name already.

**Final tool surface (~10):**

| Fleet | Orchestra |
|---|---|
| `exec` | `codex` |
| `script` | `gemini` |
| `read` | `claude` |
| `write` | `poll` |
| `transfer` | |
| `status` | |

### 2.3 Configuration Centralization

```python
@dataclass(frozen=True)
class MaestroConfig:
    issuer_url: str
    transport: str
    ssh_timeout: int = 300
    max_retries: int = 3
    agent_timeout: int = 600
    max_inline_output: int = 1500
    poll_cooldown: int = 30
    task_eviction_seconds: int = 3600
    transfer_token: str = ""
    max_transfer_size: int = 100 * 1024 * 1024
    # ... all constants in one place

    @classmethod
    def from_env(cls, transport: str) -> "MaestroConfig": ...
```

No module reads `os.environ` directly. Tests construct `MaestroConfig` with test values.

### 2.4 State Containers

```python
class FleetState:
    """Owns host connectivity and locks."""
    def __init__(self, hosts: dict[str, HostConfig]): ...

class TaskRegistry:
    """Owns task lifecycle, cooldown, eviction."""
    def __init__(self, config: MaestroConfig): ...
```

No module-level mutable globals. Instantiated once in `server.py`, passed to tool registration.

### 2.5 `maestro_status` → Structured JSON

Replace ASCII-art output with parseable JSON:
```json
{"hosts": {"apollyon": {"status": "connected", "local": true}, ...}, "available": 4, "total": 4}
```

Smaller, no wasted formatting tokens. Orchestrator formats for humans if needed.

## Part III: Client-Aware Unification

### 3.1 The Discovery

There is no separate "Maestro-Local." Claude Code connects to `http://localhost:8222/mcp` — the same process Claude.ai reaches via Cloudflare. The "split" is a network-path illusion, not a codebase split.

### 3.2 Client Classification

Cloudflare Tunnel adds `CF-Ray` to every proxied request. This is an unforgeable signal:

```python
def _classify_client(request: Request) -> str:
    if request.headers.get("cf-ray"):
        return "remote"       # Claude.ai via Cloudflare
    client_ip = request.client.host if request.client else ""
    if client_ip in ("127.0.0.1", "::1"):
        return "local"        # Claude Code on Apollyon
    if client_ip.startswith("10.42.69."):
        return "lan"          # Claude Desktop on Judas via LAN
    return "remote"           # Safe default: treat unknown as constrained
```

### 3.3 Per-Client Profiles

```python
CLIENT_PROFILES = {
    "remote": {                          # Claude.ai
        "block_timeout_exec": 15,
        "block_timeout_agent": 0,
        "poll_cooldown": 30,
        "max_inline_output": 1500,
        "expose_budget_param": False,
        "expose_timeout_param": False,
        "include_poll_hints": False,
        "instructions_verbosity": "minimal",
    },
    "local": {                           # Claude Code
        "block_timeout_exec": 20,
        "block_timeout_agent": 20,
        "poll_cooldown": 0,
        "max_inline_output": 4000,
        "expose_budget_param": True,
        "expose_timeout_param": True,
        "include_poll_hints": True,
        "instructions_verbosity": "full",
    },
    "lan": {                             # Claude Desktop via LAN
        "block_timeout_exec": 15,
        "block_timeout_agent": 0,
        "poll_cooldown": 15,
        "max_inline_output": 2000,
        "expose_budget_param": False,
        "expose_timeout_param": False,
        "include_poll_hints": False,
        "instructions_verbosity": "minimal",
    },
}
```

### 3.4 Session Context via ContextVar

```python
import contextvars

@dataclass
class ClientContext:
    classification: str
    profile: dict
    client_id: str | None = None

_client_ctx: contextvars.ContextVar[ClientContext] = contextvars.ContextVar("client_ctx")

# Set per-request in MCP handler middleware
# Read in any tool function: ctx = _client_ctx.get()
```

### 3.5 What This Enables

- Claude.ai gets lean, non-blocking, poll-cooldown-enforced behavior — impossible to cause UI hangs
- Claude Code keeps full blocking, full output, no cooldown — maximum throughput
- Adding Judas (or any new client) is a config entry, not a code change
- One server, one process, one systemd unit. Deployment stays trivial

## Implementation Roadmap

| Phase | Scope | Risk | Prerequisite |
|---|---|---|---|
| **A** | Config extraction (`MaestroConfig`) | Low | None |
| **0** | Token economy quick wins: remove `poll_with`/`log_with`, reduce `MAX_INLINE_OUTPUT`, trim instructions, add scope prefix | Low | None (parallel with A) |
| **B** | Transport extraction (`transport.py`, `local.py`) | Medium | A |
| **C** | Orchestra extraction + execute/dispatch merge | Medium | A, B |
| **G** | Client classification + `ClientContext` + profiles | Medium | A, C |
| **D** | Relay extraction | Low | B |
| **E** | Slim entry point (`server.py` → ~100 lines) | Low | B, C, D |
| **F** | Tool consolidation (20 → ~10), schema compression, rename | Medium | C, E |

## Projected Impact

| Metric | Before | After |
|---|---|---|
| UI timeout / "is it dead?" events | ~1-3 per session | 0 |
| Poll calls per 4-min task | 4-6 | 1-2 |
| Tool parameters per call (avg) | 5-7 | 2-3 |
| Schema overhead per session | ~12,000 bytes | ~3,000 bytes |
| Polling cost per task | ~1,500 tokens | ~0-400 tokens |
| Agent output bloat | ~4,000 chars/result | ~1,500 chars/result |
| Agent scope creep (wasted $) | ~30-50% of budget | ~5-10% |
| MCP tools in schema | 20 | ~10 |
| `server.py` lines | 2,195 | ~100 (entry point) |
| Spending signals leaked to Anthropic | Every dispatch | None |

## What NOT to Change

- **`_auto_promote` pattern** — elegant async/shield design. Needs param hiding + client-aware timeouts, not replacement.
- **`_orchestra_build_result`** — output-to-disk discipline is correct.
- **`oauth_rewrite.py`** — clean, well-scoped, single-responsibility.
- **ControlMaster SSH** — robust retry with exponential backoff.
- **Journal/ADR pattern** — keep writing these.
- **`maestro_oauth.py`** — already properly extracted.
- **Single systemd process** — one server, client-aware. Don't split.

## Open Questions

1. **Claude.ai frontend timeout:** Empirically unknown (~60-90s guess). Measure with `sleep N && echo done` for increasing N to find the exact ceiling for `MAX_BLOCK_TIMEOUT`.

2. **Per-project scope templates:** Should `CLAUDE.md` in each repo contain the scope prefix, or should it always come from the server?

3. **Transfer relay retry semantics:** Return 404 or 202 for a task whose result isn't ready yet? 202 is more semantic and lets retry scripts distinguish "not ready" from "doesn't exist."

4. **Dynamic tool schemas via FastMCP:** Can we conditionally strip parameters from the schema at session init time based on client classification? This would be Option A (clean schema per client) vs Option C (accept-and-ignore params from remote clients).
